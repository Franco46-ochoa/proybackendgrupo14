¿Por qué bulkInsert en vez de create() en loop?

Cada Model.create() genera un INSERT individual a la BD. Con 5.000 registros = 5.000 queries. queryInterface.bulkInsert() genera un solo INSERT con múltiples filas, reduciendo la carga de red y el tiempo de ejecución de minutos a segundos.

9 Bug detectado y corregido V3

Bug: Shadowing de variable duenoId en authController.js

Detectado durante la revisión de código del Sprint 2. Este bug estuvo presente desde Sprint 1 y afectaba silenciosamente la asignación de dueño en cada registro.

9.1. Descripción del problema

En authController.js, la función register declaraba let duenoId = null en la línea 22 (scope de la función). Dentro del bloque if (rol !== 'dueno'), en la línea 61, se declaraba const duenoId = codigo.duenoId || null, creando una nueva variable local que "sombreeaba" (shadowing) a la variable del scope superior.

9.2. Impacto

SíntomaCausa
Todos los usuarios registrados con código de invitación tenían duenoId: nullLa línea 81 usaba la variable del scope superior (siempre null), no la del bloque if
Los usuarios quedaban "huérfanos" — sin vinculación al dueño/empresaEl const dentro del bloque creaba un scope separado
No se manifestaba como error — el registro "funcionaba"Bug silencioso: solo se notaba al intentar filtrar datos por empresa

9.3. Código antes y después

// ─── ANTES (BUG) ─── let duenoId = null; // línea 22 - scope función if (rol && rol !== 'dueno') { // ...validación de código... const duenoId = codigo.duenoId || null; // línea 61 - SHADOW! crea scope local } const usuario = await Usuario.create({ duenoId: duenoId || null, // línea 81 - usa el de línea 22 (SIEMPRE null) }); // ─── DESPUÉS (CORREGIDO + RENOMBRADO V3) ─── let empresaId = null; // scope función if (rol && rol !== 'dueno') { // ...validación de código... empresaId = codigo.empresaId || null; // sin const - asigna al scope superior } const usuario = await Usuario.create({ empresaId: empresaId || null, // usa el valor correcto del código });
Valor para la defensa del TFI

Este bug es un excelente ejemplo para mostrar en la defensa: demuestra proceso de QA real, comprensión de scopes en JavaScript, y capacidad de detectar bugs silenciosos que no generan errores visibles pero rompen la lógica de negocio. Se detectó al renombrar duenoId → empresaId (Tarea V3-T2).

Leer y mapear fs.createReadStream(rutaCSV) .pipe(csv()) .on('data', (row) => { const mapped = {}; for (const [legacy, nuevo] of Object.entries(MAPEO_COLUMNAS)) { mapped[nuevo] = row[legacy] || null; } // 2. Validar if (!mapped.codigo || !mapped.nombre) { errores.push({ fila: row, motivo: 'codigo o nombre vacío' }); return; } mapped.activo = true; productos.push(mapped); }) .on('end', async () => { // 3. Deduplicar const codigos = productos.map(p => p.codigo); const existentes = await Producto.findAll({ where: { codigo: codigos } }); const codigosExistentes = new Set(existentes.map(p => p.codigo)); const nuevos = productos.filter(p => !codigosExistentes.has(p.codigo)); // 4. Carga masiva await sequelize.transaction(async (t) => { await Producto.bulkCreate(nuevos, { validate: false, transaction: t }); }); // 5. Auditoría console.log(`Importados: ${nuevos.length}, Errores: ${errores.length}`); }); }
